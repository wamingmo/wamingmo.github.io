空分复用

  • 空分复用是指在不同的时间，不同的用户或进程可以使用相同的物理资源（如内存空间）。在内存管理中，空分复用可以让多个进程共享有限的物理内存。
  • 共享页面：多个进程可以共享一些相同的物理页面。例如，操作系统中的某些库函数（如C标准库中的printf函数）是被多个进程调用的。这些库函数的代码存储在物理内存的某些页面中。通过过页表的设置，不同进程的虚拟地址空间可以映射到这些相同的物理页面。这样，物理内存中的这些页面就可以被多个进程复用。

pte结构：

  • V(Valid)：仅当位 V 为 1 时，页表项才是合法的；
  • R(Read)/W(Write)/X(eXecute)：分别控制索引到这个页表项的对应虚拟页面是否允许读/写/执行；
  • U(User)：控制索引到这个页表项的对应虚拟页面是否在 CPU 处于 U 特权级的情况下是否被允许访问；
  • G：暂且不理会；
  • A(Accessed)：处理器记录自从页表项上的这一位被清零之后，页表项的对应虚拟页面是否被访问过；
  • D(Dirty)：处理器记录自从页表项上的这一位被清零之后，页表项的对应虚拟页面是否被修改过。
  • 当 V 为 0 的时候，代表当前指针是一个空指针，无法走向下一级节点，即该页表项对应的虚拟地址范围是无效的；

  • 注意: 当 V 为1 且 R/W/X 不全为 0 时，表示是一个合法的页表项，其包含了虚地址对应的物理页号。

Trap

特权级切换的核心是对Trap的管理。这主要涉及到如下一些内容：

  • 应用程序通过 ecall 进入到内核状态时，操作系统保存被打断的应用程序的 Trap 上下文；
  • 操作系统根据Trap相关的CSR寄存器内容，完成系统调用服务的分发与处理；
  • 操作系统完成系统调用服务后，需要恢复被打断的应用程序的Trap 上下文，并通 sret 让应用程序继续执行。
  • 当一个应用 Trap 到内核时，sscratch 已指向该应用的内核栈栈顶，我们用一条指令即可从用户栈切换到内核栈.

sscratch CSR 的用途

在特权级切换的时候，我们需要将 Trap 上下文保存在内核栈上，因此需要一个寄存器暂存内核栈地址，并以它作为基地址指针来依次保存 Trap 上下文的内容。但是所有的通用寄存器都不能够用作基地址指针，因为它们都需要被保存，如果覆盖掉它们，就会影响后续应用控制流的执行。

事实上我们缺少了一个重要的中转寄存器，而 sscratch CSR 正是为此而生。从上面的汇编代码中可以看出，在保存 Trap 上下文的时候，它起到了两个作用：首先是保存了内核栈的地址，其次它可作为一个中转站让 sp （目前指向的用户栈的地址）的值可以暂时保存在 sscratch 。这样仅需一条 csrrw sp, sscratch, sp 指令（交换对 sp 和 sscratch 两个寄存器内容）就完成了从用户栈到内核栈的切换，这是一种极其精巧的实现。

一条汇编指令的解读

  1. 指令功能
  • csrrw 指令
    • csrrw 是 RISC-V 指令集中的一个指令，全称是 Control Status Register Read and Write。
    • 它的功能是读取一个控制状态寄存器的值到一个通用寄存器中，同时将该通用寄存器的值写入到该控制状态寄存器。
  • sp 寄存器
    • sp 是栈指针寄存器（Stack Pointer），通常用于指向当前栈顶的位置。
  • sscratch 寄存器
    • sscratch 是一个控制状态寄存器，通常用于保存临时数据，特别是在上下文切换时。
  1. 指令解析
  • csrrw sp, sscratch, sp
    • 这条指令的作用是：
      1. 将 sscratch 寄存器的值读取到 sp 寄存器中。
      2. 将 sp 寄存器的值写入到 sscratch 寄存器中。
    • 简单来说，它交换了 sp 和 sscratch 寄存器的值。

用户虚拟地址空间范围

子进程的地址空间

在 Linux 中，当父进程调用 fork() 创建子进程时，子进程的地址空间是通过一种高效的机制设置的。这个过程涉及以下几个关键步骤和机制：

  1. 写时复制（Copy-On-Write，COW）

fork() 的核心机制是 写时复制，它是一种延迟拷贝技术，用于高效地创建子进程。具体来说：

  • 在调用 fork() 时，子进程的地址空间（包括代码段、数据段、堆、栈等）并不会立即被完整复制一份。
  • 相反，父进程和子进程的地址空间在创建时是共享的，但每个地址空间的页面（内存块）被标记为“只读”。
  • 当父进程或子进程试图修改某个共享页面时，操作系统会触发一个“写时复制”事件：
    • 操作系统会为修改的进程创建该页面的独立副本。
    • 修改的进程会继续在新的副本上操作，而另一个进程仍然使用原来的共享页面。
  • 这种机制大大减少了 fork() 的开销，因为只有在实际需要修改内存时，才会进行复制。
  1. 地址空间的初始化

在 fork() 调用时，子进程的地址空间会按照以下方式初始化：

  • 代码段（Text Segment）：代码段通常包含程序的可执行代码，通常是只读的。在 fork() 时，代码段会被标记为共享且只读，因此父进程和子进程可以共享相同的代码段，无需复制。
  • 数据段（Data Segment）：数据段包含全局变量和静态变量。在 fork() 时，数据段的页面会被标记为“只读”，并采用写时复制机制。如果父进程或子进程试图修改数据段中的内容，操作系统会为修改的进程创建独立的页面副本。
  • 堆（Heap）：堆用于动态内存分配（如通过 malloc() 分配的内存）。在 fork() 时，堆的页面同样被标记为“只读”，并采用写时复制机制。当父进程或子进程试图修改堆中的内容时，操作系统会为修改的进程创建独立的页面副本。
  • 栈（Stack）：栈用于函数调用和局部变量存储。在 fork() 时，栈的页面也会被标记为“只读”，并采用写时复制机制。当父进程或子进程试图修改栈中的内容时，操作系统会为修改的进程创建独立的页面副本。
  1. 文件描述符和资源

除了地址空间，子进程还会继承父进程的其他资源：

  • 文件描述符：子进程会继承父进程打开的所有文件描述符。
  • 信号处理设置：子进程会继承父进程的信号处理函数和信号屏蔽集。
  • 环境变量和工作目录：子进程会继承父进程的环境变量和当前工作目录。
  1. 效率优势

写时复制机制使得 fork() 在大多数情况下非常高效。例如：

  • 如果子进程创建后立即调用 exec()（如在常见的“fork + exec”模式中），则子进程的地址空间不会被复制，因为 exec() 会替换掉子进程的地址空间。
  • 如果父进程和子进程在 fork() 后没有修改共享页面，则它们可以共享相同的物理内存，节省资源。

总结

在 fork() 时，子进程的地址空间是通过写时复制机制初始化的。父进程和子进程的地址空间在创建时是共享的，但每个页面被标记为“只读”。只有在实际需要修改页面时，操作系统才会为修改的进程创建独立的页面副本。这种机制既保证了子进程的独立性，又提高了 fork() 的效率。

load_page_fault和store_page_fault

在 Rust 中，特别是在 RISC-V 架构的上下文中，LoadPageFault 和 StorePageFault 是两种不同类型的内存访问异常，它们分别对应于加载（读取）操作和存储（写入）操作时触发的页错误。

LoadPageFault

  • 定义：当程序尝试从一个未映射的虚拟地址加载（读取）数据时，会触发 LoadPageFault。
  • 触发场景：
    • 访问的虚拟地址没有对应的页表项。
    • 访问的地址权限不足（例如，没有读权限）。
  • 处理方式：操作系统通常会捕获该异常，并根据情况决定是分配新的物理页面、修复页表，还是终止程序。

StorePageFault

  • 定义：当程序尝试向一个未映射的虚拟地址存储（写入）数据时，会触发 StorePageFault。
  • 触发场景：
    • 访问的虚拟地址没有对应的页表项。
    • 访问的地址权限不足（例如，没有写权限）。
  • 处理方式：与 LoadPageFault 类似，操作系统会捕获该异常，并根据情况决定是分配新的物理页面、修复页表，还是终止程序。

区别

  • 操作类型：
    • LoadPageFault 是在读取操作时触发的。
    • StorePageFault 是在写入操作时触发的。
  • 用途：
    • LoadPageFault 主要用于处理读取未映射或权限不足的内存。
    • StorePageFault 主要用于处理写入未映射或权限不足的内存。

futex

futex（fast userspace mutex）是 Linux 内核提供的一种“绝大多数时间工作在用户态、只有在真正需要阻塞或唤醒时才陷入内核”的同步原语，用来高效实现锁、条件变量等线程同步机制。

  1. 动机：为什么要有 futex？

传统 System V IPC（semget, semop …）或早期 POSIX 信号量/互斥体每次操作都要系统调用，代价高。
futex 把「无竞争时完全在用户态完成」和「有竞争时由内核帮忙挂起/唤醒」结合起来，兼顾正确性和性能。

  1. 工作原理（两步走）

  2. 用户态 fast-path
    线程在共享内存中放一个 32 位整数（futex word）表示锁状态；
    用原子指令（LDREX/STREX, cmpxchg…）尝试获取锁——成功就直接返回，0 syscall。

  3. 内核 slow-path（只在需要时）
    如果尝试失败（发现已被占用）：

    • 调用 futex(..., FUTEX_WAIT, expected, ...)：内核检查值是否仍为 expected，是则把当前任务挂起；
    • 持有者释放锁后调用 futex(..., FUTEX_WAKE, n, ...)：内核唤醒最多 n 个等待者。

  4. 关键系统调用

  • op 决定行为：FUTEX_WAIT, FUTEX_WAKE, FUTEX_REQUEUE, FUTEX_WAKE_OP, FUTEX_WAIT_BITSET …
  • 返回值：成功返回 0 或唤醒数；失败返回 -1 置 errno。
  1. 内核里发生了什么？
  • 挂起等待：futex_wait() → futex_wait_setup() → schedule() 把 task 置为 TASK_INTERRUPTIBLE，挂入 futex 等待队列。
  • 唤醒：futex_wake() → wake_up_q() → try_to_wake_up() 把等待者状态改为 TASK_RUNNING，放回调度队列。
  • 队列管理：每个 futex 地址在内部哈希表 futex_hash 中对应一个 futex_q 链表，锁粒度很细，避免全局大锁。
  1. 用 futex 构建高层同步器

glibc 的 pthread_mutex_t、pthread_cond_t、sem_t、C++11 的 std::mutex/std::condition_variable，以及 Rust 的 std::sync::Mutex 等，底层全部用 futex 实现。

维度 说明
作用 用户态/内核态混合的同步原语
性能 无竞争时 ≈ 原子指令；有竞争时 syscall

MapPermission-rv/la

✅ RISC-V 64 的 MapPermission

🔍 解释：

  • RISC-V 的 PTE（页表项）格式中，权限位如下：
    • bit 0: V（Valid，存在位）
    • bit 1: R（Readable）
    • bit 2: W（Writable）
    • bit 3: X（Executable）
    • bit 4: U（User accessible）

✅ LoongArch 64 的 MapPermission

🔍 解释：

  • LoongArch 的 PTE 格式与 RISC-V 不同，权限位分布更复杂，且采用了“反向权限”设计：
    • NX：禁止执行（为 1 表示不可执行）
    • NR：禁止读取（为 1 表示不可读）
    • W：可写（为 1 表示可写，不是反向）
  • PLVL/PLVH：组合成 2 位的 PLV（Privilege Level），表示该页表项对应的特权等级。
  • RPLV：Restricted PLV，若置位，则只有 PLV 完全匹配的进程才能访问该页。

✅ 总结对比表：

特性 RISC-V 64 LoongArch 64
可读 R = 1 << 1 !NR（NR=1 表示不可读）
可写 W = 1 << 2 W = 1 << 8
可执行 X = 1 << 3 !NX（NX=1 表示不可执行）
用户态访问 U = 1 << 4 由 PLV 和 RPLV 控制
特权等级控制 无（由 U 位区分） PLVL/PLVH + RPLV
权限位方向 正向（置位表示允许） 混合（NX/NR 为反向，W 为正向）

NUMA

  1. 为什么需要 NUMA 策略？
  2. Linux 到底有哪几类策略？

一、先画一张 NUMA 机器图，建立直觉

假设你有一台 2-路服务器：

  • 对 CPU 0/1 来说，访问节点 0的内存叫本地访问（100 ns）。
  • 访问节点 1 的内存叫远程访问（可能 300 ns）。

如果所有线程都随机拿内存，就会出现“CPU 0 经常访问节点 1”，性能白白下降。
NUMA 内存策略就是“让程序员告诉内核：请尽量把内存放在××节点”。

二、Linux 支持的 6 种策略（一张表背下来）

策略宏 中文名 内核行为规则（白话） 何时用
MPOL_DEFAULT 系统默认 “你在哪个 CPU 运行，就从哪个节点拿内存”。完全不管别的。 默认
MPOL_BIND 绑定 “只能在我指定的节点列表里拿；如果这几个节点全没内存，就失败”。 需要硬隔离，如实时任务
MPOL_PREFERRED 优先 “先尝试我指定的节点；如果它没内存，再去最近的其它节点”。 大多数调优场景
MPOL_LOCAL 强制本地 显式说“我就要本地节点”，优先级高于 DEFAULT，但不如 BIND 硬。 同 DEFAULT，但可覆盖父策略
MPOL_INTERLEAVE 轮询 在多个节点轮流各拿一页（0→1→0→1…）。提高总带宽，牺牲局部性。 大数组/大矩阵并行计算
MPOL_WEIGHTED_INTERLEAVE 加权轮询 上面升级版，节点权重不同，按比例轮询。 节点带宽差异大时

补充：还有一个 MPOL_PREFERRED_MANY，Linux 5.14+ 加入，允许一次指定多个“优先节点”，行为类似 PREFERRED，但候选集合更大。

三、策略可以挂在 3 个“挂载点”

  1. 线程级（thread policy）
    用 set_mempolicy() 设置，之后该线程所有新匿名页、堆、栈都服从它。
    查询：get_mempolicy(flags=0)。
  2. 地址区间级（VMA policy）
    用 mbind() 指定一段 [start, end) 地址。之后只要这段地址发生缺页，就按该策略。
    查询：get_mempolicy(MPOL_F_ADDR, addr)。
  3. 共享对象级（shared policy）
    同样用 mbind()，但目标可以是 shmget() 的共享内存、或 mmap() 的 tmpfs 文件。
    所有映射到这块物理页的进程都会受同一策略影响。

参考代码使用insert with hint

之前我仔细看参考代码的时候，发现他们alloc user res的时候，使用的是insert with hint当时我还好奇为什么这么麻烦，还得先递归寻找可用页面，再分配，根据tid直接计算分配不是更好，还节省性能。今天知道为什么了，因为在创建子进程的时候，子进程首先使用父进程的地址空间，如果子进程exec其他应用的时候再重新分配stack和trap资源，但是问题是如果直接copy父进程的stack和trap信息的话，创建子进程的时候返回会返回到父进程的trap信息处，这里似乎要返回子进程的trap信息，我们参考的代码是给子进程一个存储trap的空间，然后copy父进程的数据，之后如果exec其他应用的话，再重新给子进程分配一个trap页，所以使用的是insert with hint ，而不能直接根据tid分配，因为tid唯一，做不到给一个进程分配两个trap页。

这个结构体 ext4_blockdev 是 ext4 文件系统中用于**抽象块设备（block device）**的核心数据结构。它封装了底层块设备的接口、缓存、分区信息以及与文件系统的关联。下面逐个字段解释：

  1. bdif: *mut ext4_blockdev_iface
  • 作用：指向块设备接口（interface）的指针，定义了底层块设备的操作函数（如读写块、同步等）。
  • 用途：通过此接口，文件系统可以屏蔽不同硬件设备的差异，统一调用标准方法。
  1. part_offset: u64
  • 作用：当前分区在整个物理设备中的起始偏移（以字节为单位）。
  • 场景：当物理设备被划分为多个分区（如 /dev/sda1、/dev/sda2）时，此字段表示当前分区从物理设备的哪个字节开始。
  1. part_size: u64
  • 作用：当前分区的总大小（以字节为单位）。
  • 场景：与 part_offset 配合使用，定义了分区的边界。
  1. bc: *mut ext4_bcache
  • 作用：指向块缓存（block cache）的指针，用于缓存磁盘上的块数据以提高性能。
  • 细节：ext4 通过缓存减少直接访问磁盘的次数，支持延迟写回（writeback）或直写（writethrough）策略。
  1. lg_bsize: u32
  • 作用：逻辑块大小（logical block size），单位为字节。
  • 注意：这是文件系统层面的块大小（通常为 4KB），可能与物理设备的块大小不同。
  1. lg_bcnt: u64
  • 作用：逻辑块的总数（逻辑块的计数）。
  • 计算：分区大小除以逻辑块大小（part_size / lg_bsize）。
  1. cache_write_back: u32
  • 作用：写回模式（writeback）的引用计数。
  • 解释：
    • 当计数 > 0 时，缓存采用写回模式（数据先写入缓存，延迟刷盘）。
    • 当计数 = 0 时，可能切换为直写模式（数据立即写入磁盘）。
  • 用途：支持动态切换缓存策略（如挂载时指定 sync 或 async 模式）。
  1. fs: *mut ext4_fs
  • 作用：指向此块设备所属的 ext4 文件系统实例。
  • 关联：通过此指针可以访问文件系统的超级块、inode 表等元数据。
  1. journal: *mut c_void
  • 作用：指向日志（journal）的指针，用于支持 ext4 的日志功能（保证崩溃一致性）。
  • 细节：日志可以是独立的块设备（如外部 journal），也可以是同一设备上的特定区域。c_void 表示底层实现可能为 ext4_journal 或其他类型。

总结：结构体的角色

这个结构体是ext4 文件系统与底层存储设备之间的桥梁，通过它：

  1. 管理分区：通过 part_offset 和 part_size 支持多分区。
  2. 统一设备操作：通过 bdif 屏蔽硬件差异。
  3. 优化性能：通过 bc 缓存和 cache_write_back 策略减少磁盘访问。
  4. 支持日志：通过 journal 保证崩溃后的数据一致性。
  5. 关联文件系统：通过 fs 指针与文件系统元数据关联。

简单来说，ext4_blockdev 是 ext4 对块设备的高层抽象，所有对磁盘的读写最终都会经过这个结构体协调。

ext4_blockdev_iface

这个结构体 ext4_blockdev_iface 是 ext4 文件系统与底层块设备之间的接口抽象层，定义了操作块设备所需的所有函数指针和元数据。它相当于一个 “驱动程序接口”，让 ext4 可以适配不同的硬件设备（如磁盘、SD 卡、虚拟块设备等）。

📌 核心作用

  • 统一设备操作：通过函数指针屏蔽不同硬件的实现差异。
  • 提供设备信息：记录物理块大小、总块数等关键参数。
  • 支持并发控制：通过 lock/unlock 实现多分区模式下的互斥访问。
  • 统计与调试：通过计数器记录读写次数，方便性能分析。

🔍 字段详解

TCP/UDP

当然可以！UDP 和 TCP 是互联网传输层的两大核心协议，它们都基于 IP 协议，但设计哲学和工作原理截然不同。下面我用通俗的语言和例子帮你理解它们的原理和区别。

1. TCP（传输控制协议）：像“打电话”

  • 特点：可靠、有序、面向连接。

  • 核心原理：

    • 三次握手建立连接：
      客户端和服务器先互相确认“你能听到我吗？”（类似打电话时的“喂？能听到吗？”）。
      例：客户端发送 SYN → 服务器回复 SYN-ACK → 客户端再回 ACK，连接建立。
    • 数据分段与排序：
      数据被拆分成多个小包（如发送一张图片），每个包标记顺序号。若某个包丢失（如第3包），接收方会要求重传，最终按顺序组装还原。
    • 流量控制：
      根据接收方的处理能力调整发送速度（比如对方手机信号差时，说“慢点说”）。
    • 四次挥手断开连接：
      双方确认“我说完了，可以挂了”（如 FIN → ACK → FIN → ACK）。

  • 适用场景：网页（HTTP/HTTPS）、邮件（SMTP）、文件传输（FTP）等需要数据完整的应用。

2. UDP（用户数据报协议）：像“发短信”

  • 特点：无连接、不可靠、高效。

  • 核心原理：

    • 无需握手：直接发数据，不管对方是否在监听。
      例：你直接发短信给朋友，不确认他是否开机。
    • 无状态传输：
      每个数据包（称为“数据报”）独立发送，不标记顺序，不追踪是否到达。若丢失（如短信没收到），发送方完全不知道。
    • 无流量控制：
      以恒定速度发送，不管网络是否拥堵（类似不停群发消息）。
    • 轻量头部：
      仅8字节头部（TCP为20字节以上），传输开销小。

  • 适用场景：视频直播、在线游戏、DNS查询等需要实时性、可容忍少量丢包的应用。

对比表格：TCP vs UDP

特性 TCP UDP
连接方式 面向连接（三次握手） 无连接（直接发）
可靠性 保证数据完整、有序 不保证，可能丢包或乱序
传输效率 较低（需确认、重传、排序） 极高（无额外机制）
流量控制 有（滑动窗口机制） 无
头部大小 20+字节 8字节
典型应用 网页、邮件、文件传输 视频、语音、游戏、DNS

生活化例子

  • TCP：网购快递（必须签收、按顺序到货，丢失会补发）。
  • UDP：电台广播（信号可能断断续续，但实时性强，错过就错过了）。

总结一句话

TCP是“可靠但慢”的协议，UDP是“快速但不可靠”的协议——选择哪个取决于应用更看重数据完整性还是实时性。

sendto(buf,len,addr) 把用户数据拷到单个 sk_buff，直接填 UDP 首部 → 交给 IP 层 无流量控制，不保证到达
recvfrom() 从输入队列弹出一个 完整数据报，拷贝到用户空间；队列空 → EAGAIN 队列溢出直接丢包
close() 释放 udp_sock 和输入队列 无状态，立即回收

UDP 的“输入队列”只负责存完整报文，无顺序、无确认、无重传。

⏱️ TCP 时间线（面向连接）

阶段 内核做了什么 关键结构
socket(AF_INET, SOCK_STREAM) 创建 struct tcp_sock，准备 两个队列：发送缓冲区、接收缓冲区
bind(port) + listen() 把 tcp_sock 标为 LISTEN，放入 半连接 hash (SYN queue) 等待三次握手
accept() 阻塞直到 SYN queue 有完成握手的子 socket；返回新的 tcp_sock 子 socket 继承父 socket 的缓冲区大小
send(buf,len) 把用户数据切成 MSS 片，放入发送缓冲区 → 滑动窗口 → ACK 驱动 窗口满时阻塞或返回 EAGAIN
recv() 从接收缓冲区顺序读字节流；无数据时阻塞 乱序包先进入 乱序队列，再合并
close() 发起 四次挥手；维护 TIME_WAIT 定时器 资源在 TIME_WAIT 结束后回收

TCP 的 接收缓冲区 会重排乱序段、做流量控制；发送缓冲区 负责超时重传。

  • 触发 COW 缺页异常（cow_page_fault）。

    • 分配一个新的物理页。
    • 复制原页内容到新页。
    • 更新当前进程的页表项，指向新页，并标记为可写。

    • 父进程和子进程的虚拟地址相同。
    • 物理页不同（不再共享）。

  • 各自独立，互不干扰。
  • 即使一方释放或扩展堆，不影响另一方。

命令码 9 对应的正式名称是

FUTEX_WAKE_OP

它不是简单地把某个线程唤醒，而是

  1. 先在 uaddr2 指向的 futex 字上做一次原子运算（加、或、与、异或等），

    如果比较结果为假，就唤醒 uaddr2 上最多 val 个等待者。

典型用途：把条件变量的“等待-触发”逻辑全部塞进一次 futex 调用里完成，避免竞争。

  • op_arg1：对 uaddr2 做的运算

  • op：比较运算

#[inline(always)] 是 Rust 的一个属性（attribute），告诉编译器：请一定把这段函数体直接“粘贴”到每个调用点去，不要生成真正的函数调用指令。

为什么这么做？

  1. 减少函数调用开销：
    普通函数调用需要压栈参数、跳转、返回，这些有 CPU 周期开销。内联后，这些指令直接展开到调用处，省去了调用/返回的开销。

  2. 优化器更好发挥：
    内联后，编译器能看到调用上下文，可能会进一步优化（比如常量折叠、消除边界检查等）。

  3. always 是强制性的：
    普通的 #[inline] 只是“建议”，编译器可以拒绝。但 #[inline(always)] 是强制性的（除非遇到特殊情况，如递归或动态分发），编译器必须内联。

地址族

一句话：地址族（Address Family）就是“地址的格式说明书”。

内核里凡是出现 AF_XXX 的常量，都是在告诉你“接下来的 IP/端口/路径该怎么解析、占多少字节”。常见值：

AF_INET → IPv4 地址，4 字节 IP + 2 字节端口
AF_INET6 → IPv6 地址，16 字节 IP + 2 字节端口
AF_UNIX → 本地文件系统路径，可变长字符串
AF_PACKET → 原始链路层帧，以太网头格式

因此：

• 创建 socket 时你必须先选 AF，告诉内核“后面我要给的是哪种地址”。
• 后续所有系统调用（bind、connect、sendto、accept）都把 sockaddr 强转成对应的格式。

没有地址族，内核就无法知道 struct sockaddr 里那 14 个字节到底是 IPv4、IPv6，还是别的。

这些值只是“协议族”常量，在内核/系统库里用整数区分，没有内在算术意义——就像枚举标签。

AF_INET6 10 IPv6：16 byte IP + 2 byte port
AF_UNIX 1 Unix 域套接字：本地文件系统路径

阶段 3：数据传输（ESTABLISHED）
3.1 接收方向
• 网卡收到数据包 →
‑ 若 seq == rcv_nxt：直接追加到 rx（应用可读）
‑ 若 seq > rcv_nxt：把 RecvSegment 插入 ooo_map，等待空洞填补
‑ 触发 pending_ack = true，启动 延迟 ACK 定时器 (delack_timer.start())

• 定时器到期 → 发送 ACK，清空 pending_ack，rcv_nxt += 已交付字节

3.2 发送方向
• 用户 write(buf) → 把字节推到 tx
• 发送逻辑循环：
允许发送窗口 = min(peer_wnd, cwnd)
从 tx 弹出 ≤ MSS 的数据 → 构建 SentSegment →
‑ 填入 seq = snd_nxt，len = data.len() →
‑ snd_nxt += len →
‑ 启动 RTO 定时器 →
‑ 包放入 rtx_queue → 真正发送

3.3 收到 ACK
• 解析 ack_seq → 把所有 SentSegment.seq + len ≤ ack_seq 从 rtx_queue 删除
• 更新 snd_una = ack_seq
• 根据 RTT 样本更新 rto_timer.rtt_estimator()
• 拥塞控制：
‑ 若处于慢启动：cwnd += MSS
‑ 若到达 ssthresh：进入拥塞避免 cwnd += MSS*MSS/cwnd

3.4 超时重传
• rto_timer 触发 → 把 rtx_queue.front() 重新发送
• rtx_count += 1；若超过阈值，可 close() 并发送 RST
• 拥塞反应：ssthresh = max(cwnd/2, 2*MSS)，cwnd = MSS（TCP Tahoe）

阶段 4：四次挥手（主动/被动关闭）
• 用户 close() → state = FinWait1，放入 FIN 段到 tx 并发送
• 收到对端 FIN → 根据当前状态机迁移
‑ FinWait1 → FinWait2 → TimeWait 或
‑ CloseWait → LastAck
• TimeWait 定时器到期后真正释放资源

阶段 5：资源回收
• 所有定时器停止
• rx/tx/rtx_queue/ooo_map 清空
• state = Closed

一句话总结
TcpInner 像一个“迷你 TCP 端点”：

  • rx/tx 对接用户；
  • snd_* / rcv_* 与序号空间打交道；
  • 定时器 + 队列（rtx_queue / ooo_map / delack_timer）保证可靠、按序、流量/拥塞控制；
  • 状态机字段（state, pending_conns, completed_conns）驱动连接生命周期。

vDso

vDSO = “virtual Dynamic Shared Object” —— 内核在 用户态地址空间 里 映射的一小段只读代码/数据，用来让 libc 调用 “快”且不进入内核 的系统调用（如 gettimeofday、clock_gettime、sigreturn 等），同时避免 系统调用号/ABI 差异 带来的兼容性问题。

为什么需要 vDSO？

  1. 加速
    像 gettimeofday 这类纯读系统信息，不修改内核状态，内核提前把实现 编译成一段机器码 拷到 vDSO 里；
    用户态直接 call vDSO → 不走 syscall/ecall，省一次上下文切换。

  2. 兼容
    不同架构/内核版本的系统调用号、寄存器约定不一样，libc 只需 链接到 vDSO 提供的符号（__vdso_gettimeofday…），内核升级后 vDSO 内容跟着变，用户程序无需重编。

  3. 信号返回
    glibc 把 rt_sigreturn 的 stub 放在 vDSO 里；若内核没映射 vDSO，stub 地址就是悬空 → InstructionPageFault（你遇到的情况）。

1. DA（Direct Address）

  • DA=1 表示启用 直接地址翻译模式（Direct Address Translation Mode）。
  • 在此模式下，虚拟地址直接等于物理地址（或低位部分直接映射），不经过页表或TLB转换。
  • 处理器复位后默认进入该模式（DA=1, PG=0），此时 PC=0x1C000000 对应的物理地址也是 0x1C000000。

2. PG（Paged）

  • PG=1 表示启用 映射地址翻译模式（Paged Address Translation Mode）。
  • 在此模式下，地址翻译通过 直接映射窗口 或 页表映射（TLB） 完成。
  • 操作系统正常运行时，通常启用该模式（DA=0, PG=1），以支持虚拟内存管理。

两种模式的组合关系

DA PG 模式说明
1 0 直接地址翻译模式（复位默认）
0 1 映射地址翻译模式（启用页表）
其他组合 非法或未定义

内核/固件中的使用场景

  • 启动阶段：固件或内核启动初期使用 直接地址翻译模式（DA=1, PG=0），以简化地址管理。
  • 正常运行：操作系统初始化完成后，切换到 映射地址翻译模式（DA=0, PG=1），启用虚拟内存。
  • 从休眠唤醒：部分固件可能未在唤醒时自动启用 PG 模式，此时内核需手动设置 PG=1 以确保正常运行。

总结

  • DA=1：虚拟地址=物理地址，无地址转换，适合启动或裸机环境。
  • PG=1：启用 页表/TLB机制，支持虚拟内存，适合操作系统正常运行。

git工作流程

你的日志显示 Git 正在正常地读取 /home/.gitconfig 文件。它先以只读 (O_RDONLY) 模式打开，然后使用 fstat 系统调用多次检查文件的元数据（O_ASK_SYMLINK），这是程序在处理文件时的标准流程。

Git 配置文件的查找顺序

Git 会按照一个固定的优先级顺序来查找和加载配置文件：

  1. 仓库级 (.git/config)：优先级最高，只对当前仓库有效。
  2. 用户级 (~/.gitconfig 或 ~/.config/git)：优先级次之，对当前用户的所有仓库有效。
  3. 系统级 (/etc/gitconfig)：优先级最低，对系统上所有用户的所有仓库有效。

Git 会依次检查这些文件，找到一个就读取它的内容，然后继续检查下一个，如果发现有重复的配置项，优先级高的配置会覆盖优先级低的。

ASIC

ASIC 方式是指使用 专用集成电路（Application-Specific Integrated Circuit） 来实现一个电子系统。简单来说，就是为某个特定的应用或功能，专门设计并制造一个唯一的、定制化的芯片。

ASIC方式的特点

  1. 高度定制化：芯片的电路布局和功能是完全根据你的需求来设计的，不具备通用性。它只做一件事，而且做得非常好。
  2. 性能极高：由于ASIC是为特定任务优化的，它能够达到最高的运行速度和效率。没有通用处理器中不必要的电路，所以功耗也更低。
  3. 批量生产成本低：虽然前期的设计和制造（称为“流片”）成本极高，但一旦完成，大批量生产的单个芯片成本会非常低。
  4. 开发周期长，风险高：ASIC的开发是一个漫长而复杂的过程，从设计、验证到最终制造可能需要数月甚至数年。如果设计出现错误，整个批次的芯片都可能报废。

ASIC与其他方式的对比

理解ASIC的最好方式是将其与另外两种常见的硬件实现方式进行比较：

  • 与通用处理器（如CPU、GPU）对比
    • 通用处理器：像你的电脑CPU一样，可以运行各种不同的软件程序。它的灵活性极高，但因为需要支持多种功能，所以执行某个特定任务时，效率不如ASIC。
    • ASIC：硬件即软件。它的功能在制造时就已经固化在电路上，无法改变。但正因为如此，它在执行特定任务时速度更快、功耗更低。例如，比特币矿机就是一种典型的ASIC，它只做一件事：快速计算哈希值。
  • 与FPGA（现场可编程门阵列）对比
    • FPGA：是一种可重复编程的芯片。你可以用它来快速实现一个数字电路原型。它的灵活性介于通用处理器和ASIC之间。
    • ASIC：一旦制造完成，其功能就无法改变。但在性能、功耗和最终成本上，ASIC通常优于FPGA。FPGA常用于原型验证或小批量生产，而ASIC则用于大规模、高要求的应用。

总结

ASIC方式是一种以性能和效率为核心的硬件开发策略。它牺牲了灵活性和前期成本，以换取在特定应用领域无与伦比的性能和更低的批量成本。因此，ASIC常用于那些需要极致性能、高吞吐量或低功耗的大规模应用，比如手机处理器、AI加速芯片、网络交换机、加密货币矿机等。

PPO

基于策略函数，使用策略梯度方法进行探索，可以理解为从一个初始状态下，根据每次的探索的结果给予奖励或惩罚，从而更新到下一个期望状态。

最核心的创新在于一个裁剪机制，避免因为某次的偶然选择导致奖励过多或惩罚过多。假设一次行为的奖励为100，但是我限制的奖励范围是±50，（这个限制是一个超参数，由我们提前设定），取限制和实际奖励的min，防止一次的步子迈的太大。

同时使用熵奖励，鼓励探索，避免陷于局部最优。

如果策略的熵太低，意味着策略变得过于“确定”，它总是选择相同的动作，即使这些动作可能不是全局最优的。这会导致智能体陷入局部最优，无法发现更好的策略。这就像一个餐馆老板，一旦他发现一道菜受欢迎，他就只买这道菜，不再尝试其他新菜品，最终可能错失做大做强的机会。

通过在目标函数中添加一个正的熵项（因为我们是最大化目标函数，所以是加号），PPO鼓励策略保持一定的随机性，从而促进探索。这就像给餐馆老板一个“创新奖励”，鼓励他尝试新的食材和烹饪方法，推出新菜品，即使这些尝试不一定每次都成功

LSM-Tree

LSM-Tree（Log-Structured Merge Tree）是一种面向磁盘、写友好的分层有序存储结构，核心思想只有一句话：

“先把所有写操作变成顺序追加，再定期归并整理，用后台 I/O 换前端性能。”

一、为什么会出现

  • B+ 树每次写入都要寻道、随机写页，磁盘随机 I/O 是瓶颈。
  • 大数据/日志/时序场景写多读少，需要高吞吐、低延迟写入结构。
  • LSM-Tree 把“随机写”变成“顺序写”，写入性能比 B+ 树高 5–10 倍。

二、核心组件与数据流向

  1. MemTable（内存）

    • 跳表/红黑树等有序结构；
    • 写操作先追加到 WAL（预写日志），再插 MemTable，崩溃可恢复。

  2. Immutable MemTable

    • MemTable 满后被冻结，只读；
    • 后台线程把它顺序刷盘成 SSTable 文件（Level 0）。

  3. SSTable（磁盘）

    • 按键有序、不可变；
    • 分层管理：L0 → L1 → … → Ln，容量逐层放大 10 倍；
    • 每层内部键范围不重叠（L0 除外）。

  4. Compaction（合并）

    • 定期将相邻层 SSTable 归并；
    • 消除旧版本/删除标记，控制空间放大与读放大；
    • 策略：Leveling（每层一个文件，读快写放大） vs Tiering（每层多文件，写快读放大）。

三、典型工作流程
① 写 → 追加 WAL + 插入 MemTable
② MemTable 满 → 冻结 → 刷成 L0 SSTable（顺序写）
③ 后台 Compaction → 将 L0 合并到 L1，依此类推
④ 读 → 先 MemTable → Immutable → L0 → L1…，Bloom Filter 快速跳过不存在文件。

四、优势 vs 代价
优势

  • 写入近乎内存速度，磁盘顺序落盘；
  • 易于压缩/去重，空间利用率可调；
  • 并发简单（SSTable 只读，无就地更新锁）。

代价

  • 读放大：可能要查多层文件；需靠 Bloom Filter、索引块、缓存缓解。
  • 合并放大：后台 Compaction 占用 I/O；需限速/调度策略。
  • 空间放大：旧版本未回收前暂时冗余。

五、常见系统

  • LevelDB / RocksDB（C++）
  • Cassandra / HBase / BigTable（Java）
  • TiDB、InfluxDB、Doris、ScyllaDB 等均内置 LSM-Tree 引擎。

一句话总结
LSM-Tree = “内存顺序树 + 磁盘顺序文件 + 后台归并” 三层模型；
用顺序写换随机写，用后台合并换前台性能，成为现代写密集存储系统的事实标准。

一句话总结（快递版）
LSM-Tree = “卸货区顺序堆 + 封条区顺序搬 + 后台定期并册”，
用顺序劳动换前台速度，再用并册把空间、读速度慢慢赚回来——
所以卸货飞快，查件稍慢但可接受，仓库永不爆炸。

在 LSM-Tree 里**没有“原地修改”这个概念；
任何更新、删除、修改都等价于“再写一条新记录”，随后靠合并（Compaction）**把旧版本清理掉。

  • 顾客改地址 → 不翻旧册子，直接往当前卸货区桌子写一条“最新包裹”（相同 key，新版本）。
  • 查件时先遇到新版本就返回，旧版本永远看不到。
  1. 删除（Delete）= 写“墓碑”
  • 顾客退单 → 写一条 特殊包裹：key = 原 key，value = 墓碑标记（tombstone）。
  • 查件遇到墓碑立即返回“不存在”；合并时遇到墓碑 + 旧包裹 → 一起扔掉。
